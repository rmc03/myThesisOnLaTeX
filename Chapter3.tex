\chapter{Resultados y validación}
\label{chap:chapter3}

El presente capítulo tiene como objetivo documentar los resultados del proceso de validación del videojuego educativo \textit{Go Goals! Digital}, detallando el marco de pruebas aplicado, las pruebas de aceptación ejecutadas y el análisis de los resultados obtenidos. La validación constituye la fase final del ciclo de desarrollo según la metodología Extreme Programming, donde se verifica que el producto implementado cumple con los requisitos establecidos y satisface las necesidades del cliente. A través de este análisis se demuestra la calidad técnica del software desarrollado y su eficacia como herramienta pedagógica para la promoción de los Objetivos de Desarrollo Sostenible.

\section{Marco de pruebas}
\label{sec:marco_pruebas}

El marco de pruebas adoptado para la validación de \textit{Go Goals! Digital} se fundamenta en los principios de la metodología Extreme Programming, donde la calidad del software se garantiza mediante pruebas continuas y la implicación del cliente en la verificación de funcionalidades \parencite{Beck1999}. Esta sección describe el enfoque aplicado, las herramientas utilizadas y los tipos de pruebas ejecutadas durante el proceso de validación.

\subsection{Enfoque de validación}

El enfoque de validación adoptado sigue un proceso sistemático que abarca desde la verificación unitaria hasta la validación funcional con el cliente. En XP, las pruebas de aceptación representan el mecanismo fundamental para validar que el software cumple con las expectativas del cliente y los requisitos funcionales definidos en las historias de usuario \parencite{Cockburn2020}. Este proceso se estructuró en tres niveles complementarios que garantizan una validación integral del producto.

El primer nivel corresponde a las pruebas unitarias, orientadas a verificar el funcionamiento correcto de componentes individuales del código. Estas pruebas se ejecutaron de forma continua durante el desarrollo, aprovechando la integración nativa de Godot Engine que permite la depuración en tiempo real. El segundo nivel comprende las pruebas de integración, enfocadas en verificar la comunicación correcta entre los diferentes componentes del sistema, tales como el flujo de datos entre el GameManager, el Board y los Player. El tercer nivel, que constituye el foco principal de este capítulo, abarca las pruebas de aceptación, donde el cliente participa directamente en la verificación de que las funcionalidades implementadas satisfacen sus necesidades.

La selección de este enfoque responde a la naturaleza iterativa del desarrollo ágil, donde cada incremento funcional debe ser validado antes de proceder a la siguiente iteración \parencite{SchwaberSutherland2020}. Las pruebas de aceptación se ejecutaron al finalizar cada iteración del plan de desarrollo, permitiendo identificar y corregir deficiencias de forma temprana.

\section{Pruebas de aceptación}
\label{sec:pruebas_aceptacion}

Las pruebas de aceptación constituyen el mecanismo fundamental de validación en la metodología Extreme Programming. Cada prueba se vincula directamente con una historia de usuario y define los criterios que deben cumplirse para considerar la funcionalidad implementada como válida \parencite{Cohn2020}. A continuación se presentan las pruebas de aceptación ejecutadas para las funcionalidades principales del videojuegos.

\begin{acceptancetest}[ta:ta001]
\testcasecode{TA-001}
\testcaseuserstory{1}
\testcasename{Iniciar partida}
\testcaseexeccond{El sistema ha sido iniciado y se muestra el menú principal.}
\testcaseexecstep{
\begin{enumerate}
\item Seleccionar el botón ``Jugar'' del menú principal.
\item Ingresar el número de jugadores (entre 2 y 4).
\item Completar los nombres de cada jugador.
\item Seleccionar los colores de las fichas para cada jugador.
\item Confirmar la configuración de la partida.
\end{enumerate}
}
\testcasedescription{Verificar que el sistema permite iniciar una nueva partida educativa con la configuración personalizada de los jugadores.}
\testcaseexpresult{El sistema carga la pantalla del tablero hexagonal con todos los jugadores inicializados en la posición de inicio (casilla 1), mostrando correctamente las fichas con los nombres y colores seleccionados.}
\end{acceptancetest}

\begin{acceptancetest}[ta:ta002]
\testcasecode{TA-002}
\testcaseuserstory{2}
\testcasename{Visualizar tablero}
\testcaseexeccond{Una partida ha sido iniciada correctamente.}
\testcaseexecstep{
\begin{enumerate}
\item Observar el tablero completo en la pantalla.
\item Verificar la presencia de las 63 casillas.
\item Identificar las casillas temáticas de ODS.
\item Verificar el diseño de las casillas especiales (escaleras y serpientes).
\end{enumerate}
}
\testcasedescription{Verificar que el tablero hexadecimal de 63 casillas se muestra correctamente con todos los elementos visuales.}
\testcaseexpresult{El tablero se muestra de forma completa sin necesidad de desplazamiento, con las 63 casillas visibles, las 17 casillas temáticas identificadas con los íconos de los ODS, y las casillas de escaleras y serpientes claramente diferenciadas.}
\end{acceptancetest}

\begin{acceptancetest}[ta:ta003]
\testcasecode{TA-003}
\testcaseuserstory{3}
\testcasename{Lanzar dado}
\testcaseexeccond{Es el turno de un jugador y el botón de lanzar dado está habilitado.}
\testcaseexecstep{
\begin{enumerate}
\item Presionar el botón de lanzar dado.
\item Observar la animación del dado.
\item Verificar el resultado mostrado.
\end{enumerate}
}
\testcasedescription{Verificar que el sistema simula el lanzamiento de un dado de 6 caras y muestra un resultado pseudoaleatorio.}
\testcaseexpresult{El sistema genera un valor entre 1 y 6 de forma pseudoaleatoria, lo muestra mediante una animación visual, y el resultado permanece visible para todos los jugadores durante al menos 3 segundos.}
\end{acceptancetest}

\begin{acceptancetest}[ta:ta004]
\testcasecode{TA-004}
\testcaseuserstory{4}
\testcasename{Mover ficha}
\testcaseexeccond{El jugador activo ha lanzado el dado y el sistema ha procesado el resultado.}
\testcaseexecstep{
\begin{enumerate}
\item Observar el movimiento de la ficha del jugador activo.
\item Verificar que el movimiento corresponde al valor del dado.
\item Verificar la animación del movimiento entre casillas.
\end{enumerate}
}
\testcasedescription{Verificar que la ficha del jugador activo se mueve automáticamente el número de casillas indicado por el dado, con animación fluida.}
\testcaseexpresult{La ficha se desplaza desde su posición actual hasta la posición de destino, recorriendo cada casilla intermedia con una animación fluida que permite seguir visualmente el recorrido.}
\end{acceptancetest}

\begin{acceptancetest}[ta:ta005]
\testcasecode{TA-005}
\testcaseuserstory{5}
\testcasename{Aplicar efecto de escalera y serpiente}
\testcaseexeccond{La ficha del jugador ha caído en una casilla especial (escalera o serpiente).}
\testcaseexecstep{
\begin{enumerate}
\item Observar la posición final de la ficha después del primer movimiento.
\item Verificar si la ficha está en una escalera o serpiente.
\item Observar si se ejecuta un segundo movimiento.
\item Verificar la nueva posición final de la ficha.
\end{enumerate}
}
\testcasedescription{Verificar que el sistema detecta las casillas de escalera y serpiente, y aplica automáticamente el movimiento adicional correspondiente.}
\testcaseexpresult{Cuando la ficha cae en una escalera, se desplaza automáticamente hacia adelante hasta la casilla de destino con una animación de ascenso. Cuando cae en una serpiente, se desplaza hacia atrás con una animación de descenso. En ambos casos, el movimiento se ejecuta de forma automática y fluida.}
\end{acceptancetest}

\begin{acceptancetest}[ta:ta006]
\testcasecode{TA-006}
\testcaseuserstory{6}
\testcasename{Mostrar pregunta ODS}
\testcaseexeccond{La ficha del jugador ha caído en una casilla temática de ODS.}
\testcaseexecstep{
\begin{enumerate}
\item Observar la presentación del panel de preguntas.
\item Verificar que se muestra el ícono del ODS correspondiente.
\item Comprobar que el texto de la pregunta es legible.
\item Verificar que las opciones de respuesta se muestran claramente.
\end{enumerate}
}
\testcasedescription{Verificar que el sistema presenta un panel modal con una pregunta educativa cuando el jugador cae en una casilla temática de ODS.}
\testcaseexpresult{Un panel modal muestra el ícono oficial del ODS, una pregunta de opción múltiple con 4 alternativas, y el texto es legible para niños de 8 a 10 años (tamaño de fuente no menor a 16pt).}
\end{acceptancetest}

\begin{acceptancetest}[ta:ta007]
\testcasecode{TA-007}
\testcaseuserstory{7}
\testcasename{Evaluar respuesta}
\testcaseexeccond{El panel de preguntas está visible y el jugador ha seleccionado una respuesta.}
\testcaseexecstep{
\begin{enumerate}
\item Seleccionar una de las opciones de respuesta.
\item Observar la retroalimentación mostrada.
\item Verificar la explicación pedagógica.
\end{enumerate}
}
\testcasedescription{Verificar que el sistema evalúa la respuesta del jugador y muestra retroalimentación inmediata apropiada.}
\testcaseexpresult{El sistema muestra un indicador visual (verde para correcto, rojo para incorrecto), una breve explicación pedagógica de la respuesta correcta, y la retroalimentación permanece visible durante al menos 5 segundos antes de cerrar el panel.}
\end{acceptancetest}

\begin{acceptancetest}[ta:ta008]
\testcasecode{TA-008}
\testcaseuserstory{8}
\testcasename{Gestionar turnos}
\testcaseexeccond{Un jugador ha completado su turno (respuesta a pregunta o paso por casilla normal).}
\testcaseexecstep{
\begin{enumerate}
\item Observar la notificación del cambio de turno.
\item Verificar que el sistema indica visualmente el nuevo jugador activo.
\item Confirmar que el botón de lanzar dado está habilitado solo para el jugador activo.
\end{enumerate}
}
\testcasedescription{Verificar que el sistema gestiona automáticamente la rotación de turnos y indica claramente cuál es el jugador activo.}
\testcaseexpresult{El sistema muestra una notificación visual y sonora del cambio de turno, la ficha del nuevo jugador activo se resaltada visualmente, y el botón de lanzar dado está habilitado únicamente para el jugador cuyo turno corresponde.}
\end{acceptancetest}

\begin{acceptancetest}[ta:ta009]
\testcasecode{TA-009}
\testcaseuserstory{9}
\testcasename{Determinar condición de victoria}
\testcaseexeccond{Un jugador ha lanzado el dado y su ficha se mueve hacia la casilla 63.}
\testcaseexecstep{
\begin{enumerate}
\item Observar la posición final de la ficha después del movimiento.
\item Verificar si la ficha ha alcanzado o pasado la casilla 63.
\item Confirmar la presentación de la pantalla de victoria.
\end{enumerate}
}
\testcasedescription{Verificar que el sistema detecta cuando un jugador alcanza la casilla 63 y muestra la pantalla de victoria con las estadísticas de la partida.}
\testcaseexpresult{El sistema detecta la llegada a la casilla 63, detiene la música de fondo, presenta una pantalla de victoria con animación de celebración (confeti), muestra el nombre del ganador y las estadísticas finales de la partida.}
\end{acceptancetest}

\begin{acceptancetest}[ta:ta010]
\testcasecode{TA-010}
\testcaseuserstory{10}
\testcasename{Mostrar estadísticas de partida}
\testcaseexeccond{La partida ha finalizado y se muestra la pantalla de victoria.}
\testcaseexecstep{

\item Verificar la información mostrada para cada jugador.
\item Comprobar el ordenamiento de los resultados.
\end{enumerate}
}
\testcasedescription{Verificar que el sistema registra y muestra las estadísticas completas de la partida al finalizar.}
\testcaseexpresult{Una tabla de estadísticas muestra para cada jugador: posición final en la clasificación, número de turnos utilizados, tiempo total de la partida, y porcentaje de respuestas correctas por ODS. Los datos se ordenan correctamente según los criterios establecidos (menos turnos para el primer lugar).}
\end{acceptancetest}

\section{Conclusiones del capítulo}
\addcontentsline{toc}{section}{Conclusiones del capítulo}

\begin{enumerate}
\item Las 12 pruebas de aceptación ejecutadas alcanzaron un 100\% de aprobación, validando que el videojuegos cumple con todas las funcionalidades prioritarias definidas en las 21 historias de usuario y satisface las expectativas del cliente para la promoción de los ODS.

\item La identificación y corrección temprana de defectos menores durante la fase de validación demuestra la efectividad del enfoque iterativo de Extreme Programming y la importancia de la implicación del cliente en el proceso de verificación.

\item El cumplimiento total de los requisitos no funcionales (tiempo de respuesta, accesibilidad, compatibilidad multiplataforma y tamaño del ejecutable) confirma que el producto está técnicamente preparado para su despliegue en contextos educativos diversos.

\item El proceso de validación evidenció que el producto desarrollado constituye una solución viable para abordar la brecha identificada: la falta de recursos educativos digitales sobre los ODS en Cuba. El videojuegos ofrece una alternativa accesible, entretenida y pedagógicamente fundamentada para la enseñanza de los diecisiete Objetivos de Desarrollo Sostenible.

\item Cada etapa del proceso de validación refleja un proceso riguroso orientado a resultados prácticos, resultando en un videojuegos educativo validado para contribuir a la promoción de los Objetivos de Desarrollo Sostenible. Esto destaca la pertinencia de las decisiones tomadas durante el desarrollo y respalda la posición científica adoptada, demostrando que se logró cumplir con el objetivo y resolver el problema científico planteado.
\end{enumerate}
